# Модель базы данных VPN-Service

## Назначение базы данных

База данных VPN-Service предназначена для хранения и обработки информации, необходимой для работы VPN-сервиса. Она обеспечивает:

- хранение учетных записей пользователей;
- хранение ролей и прав доступа;
- управление тарифами и подписками;
- фиксацию платежей и чеков;
- хранение сведений о VPN-серверах;
- учет подключений пользователей;
- обработку обращений в техническую поддержку;
- отправку уведомлений;
- поддержание целостности и согласованности данных между всеми модулями системы.

Основной СУБД является PostgreSQL. Структура БД спроектирована в нормализованной форме, чтобы избежать избыточности, дублирования информации и аномалий при обновлении данных.

---

## Общая логика модели

Модель данных строится вокруг сущности **user**, так как пользователь является центральным объектом системы.  
С пользователем связаны:

- его роль;
- его подписка;
- его платежи;
- его уведомления;
- его обращения в поддержку;
- его подключения к VPN-серверам.

Отдельно выделены сущности тарифов, серверов, чеков и ответов поддержки, что позволяет разделить ответственность между таблицами и сделать структуру БД расширяемой.

## Сущности базы данных

## 1. Таблица `role`

### Назначение
Таблица хранит роли пользователей в системе.  
Роль определяет уровень доступа и набор доступных действий.

### Основные атрибуты
- `id_role` — уникальный идентификатор роли.
- `role_name` — название роли.
- `description` — описание роли.
- `access_level` — числовой или логический уровень доступа.

### Использование
Таблица используется при авторизации и разграничении прав.  
Например:
- обычный пользователь может подключаться к VPN и оплачивать подписку;
- модератор работает с обращениями;
- администратор управляет пользователями, тарифами и серверами.

### Причина выделения в отдельную таблицу
Если бы роль хранилась текстом прямо у пользователя, это усложняло бы изменение прав и сопровождение системы. Отдельная таблица позволяет централизованно управлять ролями.

---

## 2. Таблица `user`

### Назначение
Таблица содержит основные сведения о пользователях системы.

### Основные атрибуты
- `id_user` — уникальный идентификатор пользователя.
- `id_role` — ссылка на роль пользователя.
- `id_subscription` — ссылка на активную подписку пользователя.
- `email` — адрес электронной почты, используемый для входа.
- `password_hash` — хеш пароля.
- `balance` — текущий баланс пользователя.
- `full_name` — имя пользователя.
- `is_active` — статус активности учетной записи.
- `created_at` — дата создания аккаунта.

### Использование атрибутов
- `email` нужен для регистрации, входа, восстановления доступа и уведомлений.
- `password_hash` используется Auth Service для безопасной проверки пароля.
- `id_role` нужен для разграничения доступа.
- `balance` используется Billing Service при оплате услуг и пополнении.
- `is_active` позволяет блокировать пользователя без удаления учетной записи.
- `created_at` используется для аудита, отчетности и отображения даты регистрации.
- `id_subscription` позволяет быстро определить текущий статус доступа пользователя к VPN.

### Особенности
Пароль хранится не в открытом виде, а в виде хеша, что соответствует требованиям безопасности.

---

## 3. Таблица `tariff_type`

### Назначение
Таблица хранит типы тарифов.

### Основные атрибуты
- `id_tariff_type` — уникальный идентификатор типа тарифа.
- `name` — название типа.
- `description` — описание типа.
- `duration` — продолжительность действия.
- `price` — стоимость.

### Использование
Эта таблица позволяет хранить общие параметры тарифных предложений.  
Например:
- месячный тариф;
- квартальный тариф;
- годовой тариф.

### Зачем нужна отдельно
Выделение типа тарифа в отдельную сущность помогает избежать повторения одних и тех же характеристик в нескольких тарифных планах и упрощает администрирование.

---

## 4. Таблица `tariff_plan`

### Назначение
Таблица хранит конкретные тарифные планы, доступные пользователям.

### Основные атрибуты
- `id_tariff_plan` — уникальный идентификатор тарифного плана.
- `id_tariff_type` — ссылка на тип тарифа.
- `name` — название тарифа.
- `max_devices` — максимальное количество устройств.

### Использование
Тарифный план используется:
- при отображении доступных подписок пользователю;
- при покупке подписки;
- при определении ограничений на количество устройств.

### Почему это важно
В системе могут существовать тарифы с разными условиями. Например, два тарифа могут иметь одинаковый срок, но разное число поддерживаемых устройств.

---

## 5. Таблица `subscription`

### Назначение
Таблица хранит сведения о подписках пользователей.

### Основные атрибуты
- `id_subscription` — уникальный идентификатор подписки.
- `id_tariff_plan` — ссылка на тарифный план.
- `start_date` — дата начала подписки.
- `end_date` — дата окончания подписки.
- `status` — текущий статус подписки.
- `auto_renew` — признак автопродления.
- `count_devices` — количество подключенных устройств.

### Использование атрибутов
- `start_date` и `end_date` позволяют определить, действует ли подписка сейчас.
- `status` нужен для проверки доступа к VPN.
- `auto_renew` используется биллингом для продления.
- `count_devices` помогает контролировать лимиты тарифа.

### Примеры статусов
- `active`
- `expired`
- `paused`
- `cancelled`

### Почему подписка выделена отдельно
Подписка — это изменяемый объект с собственным жизненным циклом.  
Она не должна храниться просто набором полей в таблице `user`, потому что:
- срок подписки меняется;
- статус меняется;
- подписка зависит от тарифа;
- возможна история изменений и расширение логики в будущем.

---

## 6. Таблица `payment`

### Назначение
Таблица хранит информацию обо всех платежах пользователя.

### Основные атрибуты
- `id_payment` — уникальный идентификатор платежа.
- `id_user` — ссылка на пользователя.
- `id_subscription` — ссылка на подписку.
- `amount` — сумма платежа.
- `payment_method` — способ оплаты.
- `status` — статус платежа.
- `created_at` — дата и время создания платежа.
- `currency` — валюта.

### Использование атрибутов
- `amount` используется для контроля корректности оплаты.
- `payment_method` нужен для отображения пользователю и интеграции с платежным шлюзом.
- `status` показывает текущее состояние операции.
- `created_at` нужен для истории транзакций и отчетности.
- `currency` позволяет поддерживать оплату в разных валютах.

### Примеры статусов
- `pending`
- `paid`
- `failed`
- `refunded`

### Роль в системе
Таблица используется Billing Service для:
- фиксации платежных операций;
- проверки успешности оплаты;
- обновления статуса подписки;
- формирования чеков.

---

## 7. Таблица `receipt`

### Назначение
Таблица хранит электронные чеки, связанные с оплатой.

### Основные атрибуты
- `id_receipt` — уникальный идентификатор чека.
- `id_payment` — ссылка на платеж.
- `receipt_number` — номер чека.
- `amount` — сумма по чеку.
- `issue_date` — дата выдачи.
- `document_name` — имя файла.
- `pdf_url` — ссылка на PDF-файл чека.

### Использование
После успешной оплаты создается чек, который:
- отображается пользователю;
- отправляется на email;
- хранится для подтверждения оплаты.

### Почему чек выделен в отдельную таблицу
Один платеж связан с юридически значимым документом.  
Чек имеет собственные атрибуты и должен храниться отдельно для удобства учета и интеграции с внешними сервисами.

---

## 8. Таблица `server`

### Назначение
Таблица хранит сведения о VPN-серверах.

### Основные атрибуты
- `id_server` — уникальный идентификатор сервера.
- `name` — название сервера.
- `location` — географическое расположение.
- `ip_address` — IP-адрес сервера.
- `current_load` — текущая нагрузка.
- `is_available` — признак доступности.

### Использование атрибутов
- `location` отображается пользователю при выборе сервера.
- `ip_address` используется для технического подключения.
- `current_load` нужен для выбора наименее загруженного сервера.
- `is_available` позволяет исключать недоступные узлы из списка.

### Роль в системе
Server Management Service использует эту таблицу для:
- мониторинга серверов;
- распределения нагрузки;
- подключения новых узлов;
- исключения неисправных серверов из работы.

---

## 9. Таблица `connections`

### Назначение
Таблица фиксирует факты подключения пользователей к VPN-серверам.

### Основные атрибуты
- `id_connection` — уникальный идентификатор подключения.
- `id_subscription` — ссылка на подписку.
- `id_server` — ссылка на сервер.
- `connected_at` — время подключения.
- `disconnected_at` — время отключения.
- `data_transferred_bytes` — объем переданных данных.
- `connection_status` — статус соединения.
- `service` — используемый сервис или протокол.
- `duration_seconds` — длительность соединения.

### Использование атрибутов
- `connected_at` и `disconnected_at` позволяют вычислить длительность сессии.
- `data_transferred_bytes` используется для статистики и анализа нагрузки.
- `connection_status` позволяет отслеживать успешные и неуспешные соединения.
- `duration_seconds` может использоваться для отчетов и мониторинга.
- `id_server` показывает, через какой сервер шло соединение.

### Зачем нужна таблица подключений
Она нужна не только для отображения истории, но и для:
- анализа нагрузки на инфраструктуру;
- поиска технических проблем;
- аудита действий;
- статистики использования сервиса.

---

## 10. Таблица `support_ticket`

### Назначение
Таблица хранит обращения пользователей в техническую поддержку.

### Основные атрибуты
- `id_support_ticket` — уникальный идентификатор обращения.
- `id_user` — ссылка на пользователя.
- `topic` — тема обращения.
- `description` — описание проблемы.
- `status` — статус обращения.
- `rejection_reason` — причина отклонения, если есть.
- `created_at` — дата создания.
- `updated_at` — дата последнего изменения.
- `priority` — приоритет обращения.
- `closed_at` — дата закрытия.

### Использование атрибутов
- `topic` и `description` нужны для содержания обращения.
- `status` позволяет отслеживать жизненный цикл тикета.
- `priority` помогает сортировать обращения.
- `rejection_reason` нужен при отклонении.
- `created_at`, `updated_at`, `closed_at` нужны для аудита и анализа скорости обработки.

### Примеры статусов
- `new`
- `in_progress`
- `resolved`
- `rejected`
- `closed`

---

## 11. Таблица `support_reply`

### Назначение
Таблица хранит ответы на обращения в поддержку.

### Основные атрибуты
- `id_support_reply` — уникальный идентификатор ответа.
- `id_support_ticket` — ссылка на обращение.
- `message` — текст ответа.
- `operator_name` — имя сотрудника или модератора.
- `created_at` — дата создания ответа.

### Использование
Таблица позволяет хранить не только одно итоговое решение, но и всю историю коммуникации по обращению.

### Почему нужна отдельно
Один тикет может содержать несколько ответов.  
Поэтому ответ нельзя хранить прямо в таблице тикетов одним полем.

---

## 12. Таблица `notification`

### Назначение
Таблица хранит уведомления, отправляемые пользователям.

### Основные атрибуты
- `id_notification` — уникальный идентификатор уведомления.
- `id_user` — ссылка на пользователя.
- `type` — тип уведомления.
- `title` — заголовок.
- `content` — текст уведомления.
- `sent_at` — дата и время отправки.

### Использование
Уведомления применяются для:
- подтверждения оплаты;
- информирования об окончании подписки;
- уведомления об ответе поддержки;
- оповещения об изменениях в системе.

### Примеры типов
- `payment_success`
- `subscription_expired`
- `support_reply`
- `system_message`

## Связи между таблицами

## Основные связи

- `role 1 — N user`  
  Одна роль может быть назначена многим пользователям.

- `user 1 — N payment`  
  Один пользователь может совершить много платежей.

- `user 1 — N support_ticket`  
  Один пользователь может создать много обращений.

- `user 1 — N notification`  
  Один пользователь может получить много уведомлений.

- `tariff_type 1 — N tariff_plan`  
  Один тип тарифа может использоваться в нескольких тарифных планах.

- `tariff_plan 1 — N subscription`  
  Один тарифный план может быть выбран многими пользователями.

- `subscription 1 — N connections`  
  В рамках одной подписки возможно множество подключений.

- `server 1 — N connections`  
  Один сервер обслуживает множество подключений.

- `payment 1 — 1 receipt`  
  Один успешный платеж формирует один чек.

- `support_ticket 1 — N support_reply`  
  Одно обращение может содержать несколько ответов.

## Как данные используются в работе системы

## Сценарий 1. Регистрация пользователя
1. Создается запись в `user`.
2. Пользователю назначается роль из `role`.
3. Пароль сохраняется в виде `password_hash`.
4. Дата регистрации фиксируется в `created_at`.

## Сценарий 2. Покупка подписки
1. Пользователь выбирает тариф из `tariff_plan`.
2. Создается запись в `payment`.
3. После успешной оплаты создается запись в `subscription`.
4. Формируется запись в `receipt`.
5. Пользователю отправляется уведомление через `notification`.

## Сценарий 3. Подключение к VPN
1. Система проверяет активность подписки в `subscription`.
2. Выбирается доступный сервер из `server`.
3. Создается запись о подключении в `connections`.
4. После завершения сессии обновляются длительность и объем данных.

## Сценарий 4. Обращение в поддержку
1. Пользователь создает тикет в `support_ticket`.
2. Модератор или оператор пишет ответ в `support_reply`.
3. Пользователь получает уведомление через `notification`.
4. Статус тикета обновляется.

## Обеспечение целостности данных

Для обеспечения корректной работы базы данных используются:

- первичные ключи для уникальной идентификации записей;
- внешние ключи для связи между таблицами;
- ограничения целостности;
- нормализация данных;
- контроль обязательных полей;
- логическое разделение сущностей по назначению.

Такой подход позволяет:
- избежать дублирования информации;
- исключить противоречивые данные;
- обеспечить надежную работу бизнес-логики;
- упростить сопровождение и масштабирование проекта.

---

## Вывод

Модель базы данных VPN-Service охватывает все ключевые сущности предметной области: пользователей, роли, тарифы, подписки, платежи, чеки, серверы, подключения, обращения в поддержку и уведомления.

Каждая таблица содержит атрибуты, непосредственно используемые в бизнес-процессах системы.  
Атрибуты не являются формальными: они участвуют в регистрации, авторизации, биллинге, подключении к VPN, поддержке пользователей, мониторинге и администрировании сервиса.

Разделение данных на отдельные сущности делает модель логически целостной, масштабируемой и удобной для дальнейшей реализации.
